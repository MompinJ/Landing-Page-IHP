Networking - segmentacion

VNet con cuatro subnets

Una sola VNet (10.0.0.0/16) dividida por funcion. Cada capa vive en su subnet y los datos quedan completamente fuera de la red publica.

vnet-hubhph - 10.0.0.0/16
10.0.1.0/24 subnet-acceso

Application Gateway v2 (entrada HTTPS publica)

10.0.2.0/24 subnet-computo

App Services con VNet Integration delegada a Microsoft.Web/serverFarms

10.0.3.0/24 subnet-datos

PostgreSQL delegado + Private Endpoints (Redis en prod)

10.0.4.0/24 subnet-soporte

Private Endpoints de Key Vault y ACR

Trafico saliente App Services salen a datos por la VNet, nunca por internet.
PE policies privateEndpointNetworkPolicies = Disabled en datos y soporte.
Entrada Front Door -> Application Gateway -> Backend pools (HTTPS 443).

Seguridad e identidad

Cero secretos en codigo

Cada App Service y Function corre con Managed Identity. Las credenciales se resuelven contra Key Vault, las imagenes vienen de un ACR sin admin user.

KV

Key Vault

kv-hubhph

Standard, RBAC habilitado. publicNetworkAccess = Disabled. Las apps acceden solo via Private Endpoint desde subnet-soporte. Aqui vive el password de PostgreSQL.

RBAC Private Endpoint Deny default
CR

Container Registry

acr-hubhph

Basic en MVP, Standard en produccion. adminUserEnabled = false: pull autenticado via Managed Identity. En produccion el endpoint publico queda apagado.

Managed Identity PE en prod
PE PostgreSQL groupId postgresqlServer en subnet-datos.
PE Key Vault groupId vault en subnet-soporte.
PE condicionales Redis y ACR solo crean PE en ambiente produccion.

Configuracion por ambiente
